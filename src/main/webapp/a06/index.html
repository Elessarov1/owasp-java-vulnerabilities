OWASP A06

Insecure Design — Missing Workflow Validation

This page demonstrates why validating an enum value is not enough: the server must also enforce which order state transitions are allowed.

Demo Scenario

Order ORDER-1001 starts in CREATED
A refund is meaningful only after payment. Try requesting REFUNDED directly while the order is still CREATED.
Vulnerable flow
The requested value is a valid enum, so the service writes it directly. The impossible transition CREATED → REFUNDED succeeds.
Fixed flow
A server-side state machine defines valid transitions. From CREATED, only PAID and CANCELLED are accepted.
Client chooses the next domain state
Enum validation checks syntax, not business meaning
Unexpected workflow states become persistent
The server owns the transition graph
Invalid transitions return 409 Conflict
State remains unchanged after rejection
The valid delivery path is CREATED → PAID → SHIPPED → DELIVERED. A paid order can also transition to REFUNDED.
