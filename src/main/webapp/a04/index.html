OWASP A04

Cryptographic Failures — Weak Password Hashing

This page demonstrates why fast unsalted hashes such as SHA-256 are not suitable for password storage, and how BCrypt improves password hashing with salt and cost.

Demo Scenario

Password Registration
The vulnerable endpoint stores passwords using SHA-256. The fixed endpoint stores passwords using BCrypt.
Vulnerable flow
The same password always produces the same SHA-256 hash. This makes leaked password hashes easier to attack offline.
Fixed flow
BCrypt stores a random salt and cost inside the final hash string. The same password produces different hashes.
SHA-256 is too fast for password storage
No work factor slows down brute force attempts
No unique salt means identical passwords produce identical hashes
BCrypt is designed for password storage
BCrypt includes a configurable cost parameter
BCrypt stores salt and cost inside the hash string
The vulnerable example is intentionally insecure. The goal is to show that cryptographic algorithms must be used for the right purpose: SHA-256 is a fast hash, not a password hashing algorithm.
